## Pipeline overview

The v0.8 flow chains four already-named methodologies into a single
narrative: a macro print arrives, we quantify the surprise, we replay it
through historical market data to see what moved, we measure cross-asset
connectedness during that window, and finally we propagate the resulting
shock through a bank-claims network to score systemic amplification.

```
   +---------------------------+
   |  Stage 1                  |
   |  Score the macro print    |
   |  cbsrm.macro.macro_events |
   |  score_event(...)         |
   +-------------+-------------+
                 |  surprise_z, direction,
                 |  risk_bias, severity
                 v
   +-----------------------------------+
   |  Stage 2                          |
   |  Replay event across markets      |
   |  cbsrm.diagnostics.macro_replay   |
   |  replay_macro_events(events,      |
   |                      prices)      |
   +-------------+---------------------+
                 |  per-event reaction
                 |  windows (returns)
                 v
   +-----------------------------------+
   |  Stage 3                          |
   |  Cross-asset connectedness        |
   |  cbsrm.indicators.dy_spillover    |
   |  DYSpilloverIndicator.compute()   |
   +-------------+---------------------+
                 |  total spillover S,
                 |  GFEVD matrix theta
                 v
   +-----------------------------------+
   |  Stage 4                          |
   |  Propagate through bank network   |
   |  cbsrm.networks.debt_rank         |
   |  debt_rank(L, E, h0)              |
   +-----------------------------------+
                 |  per-bank impact,
                 |  systemic DebtRank R
                 v
            decision intelligence
```

Each stage is independently usable. The composition is a research
narrative, not a hard dependency.

## Step 1 — Score the macro print

`cbsrm.macro.macro_events.score_event` turns a single release (actual vs
consensus, optionally with prior surprise history) into a normalised
surprise record. It is a pure function: no I/O, no persistence, no
network. The caller supplies `actual` and `consensus` from any release
calendar (BLS, BEA, Federal Reserve, FOMC OIS expectation, etc.).

```python
from cbsrm.macro.macro_events import score_event

result = score_event(
    event="CPI",
    actual=3.4,        # released headline CPI YoY %
    consensus=3.2,     # pre-release survey median
    previous=3.1,      # prior month's actual
    unit="yoy_pct",
)

print(result["surprise_z"], result["direction"], result["risk_bias"])
# 1.333... hotter_than_expected rates_up_equities_down
```

Example output (abridged):

```python
{
    "event": "CPI",
    "actual": 3.4,
    "consensus": 3.2,
    "previous": 3.1,
    "unit": "yoy_pct",
    "surprise": 0.2,
    "surprise_z": 1.3333,
    "abs_z": 1.3333,
    "direction": "hotter_than_expected",
    "severity": "moderate",
    "risk_bias": "rates_up_equities_down",

    "scale_source": "default",
    "n_history": 0,
    "spec": { "default_unit": "yoy_pct", "default_scale": 0.15, ... },

```

If the caller supplies a non-trivial `history` of prior surprises, the
sample std of `history` is used for z-scoring and `scale_source` flips to
`"history"`. The 12-event registry covers CPI / CORE_CPI / PCE / CORE_PCE /
NFP / UNRATE / INITIAL_CLAIMS / GDP / RETAIL_SALES / ISM_MANUFACTURING /
ISM_SERVICES / FOMC_RATE.

## Step 2 — Replay the event across markets

Stage 2 takes a DataFrame of scored events and a price panel and returns,
for each event, the post-print reaction window across all assets. This is
the empirical bridge between the *scoring* surface (stage 1) and the
*connectedness* surface (stage 3).

> Forward-looking: `cbsrm.diagnostics.macro_replay` is being built on
> **Lane A** and is not yet on this branch. The code block below shows
> the planned API.

```python
import pandas as pd
from cbsrm.diagnostics.macro_replay import replay_macro_events

events_df = pd.DataFrame([
    {"event": "CPI",  "date": "2026-01-14", "surprise_z":  1.33, "direction": "hotter_than_expected"},
    {"event": "NFP",  "date": "2026-02-07", "surprise_z": -0.92, "direction": "cooler_than_expected"},
    {"event": "FOMC", "date": "2026-03-19", "surprise_z":  2.10, "direction": "hotter_than_expected"},
])

prices_df = pd.read_csv("xlf_xlk_xle_xlu_tlt_dxy.csv", parse_dates=["date"], index_col="date")

report = replay_macro_events(events_df, prices_df, pre_days=5, post_days=10)
# report is keyed by (event, date); values are per-asset reaction windows
print(report.loc[("CPI", "2026-01-14")].tail())
```

The replay report feeds stage 3 by exposing the per-event return panels
on which a Diebold-Yilmaz spillover index is most informative (post-shock
windows historically show elevated connectedness).

## Step 3 — Measure cross-asset connectedness

`cbsrm.indicators.dy_spillover.DYSpilloverIndicator` computes the
Diebold-Yilmaz (2012) total spillover index over any N-column return
panel. The implementation uses the Pesaran-Shin (1998) generalized
variance decomposition, so the index is invariant to variable ordering.

```python
import pandas as pd
from cbsrm.indicators.dy_spillover import DYSpilloverIndicator

returns = pd.read_csv(
    "xlf_xlk_xle_xlu_returns.csv",
    parse_dates=["date"], index_col="date",
)

indicator = DYSpilloverIndicator(lag_order=4, forecast_horizon=10)
res = indicator.compute(returns)

print(res.values.iloc[-1])
# 62.7      <- total spillover index, %
print(res.metadata["classification"])
# 'ELEVATED_COUPLING'
print(res.metadata["fevd_matrix"])
# [[...], [...], ...]   <- N x N row-normalised GFEVD
```

For a rolling-window time series of the index, use the companion helper:

```python
from cbsrm.indicators.dy_spillover import spillover_series
series = spillover_series(returns, window=200, lag_order=4, forecast_horizon=10)
```

Classification bands (in `metadata["classification"]`): `LOW_COUPLING`
(< 25), `MODERATE_COUPLING` (25-50), `ELEVATED_COUPLING` (50-75),
`HIGH_CONTAGION` (>= 75).

## Step 4 — Propagate through the bank network

Stage 4 takes the spillover-shock implied by stage 3 (or any operator-
supplied initial-shock vector) and propagates it through a bank-claims
network using the Battiston et al. DebtRank algorithm.

> Forward-looking: `cbsrm.networks.debt_rank` is being built on **Lane B**
> and is not yet on this branch. The code block below shows the planned
> API.

```python
import numpy as np
from cbsrm.networks.debt_rank import debt_rank

# L: (N, N) interbank-exposure matrix; L[i, j] = bank i's claim on bank j
L = np.array([
    [   0.0,  20.0,  10.0,   0.0],
    [  15.0,   0.0,   5.0,  10.0],
    [   0.0,   8.0,   0.0,  12.0],
    [   5.0,   0.0,  18.0,   0.0],
])

# E: per-bank equity (capital cushion)
E = np.array([12.0, 18.0, 10.0, 16.0])

# h0: initial distress vector in [0, 1]; here a 30% shock to bank 0
h0 = np.array([0.30, 0.0, 0.0, 0.0])

result = debt_rank(L, E, h0)
print(result["debt_rank"])        # systemic R in [0, 1]
print(result["per_bank_impact"])  # final h_i per bank
```

A high `debt_rank` value means the seed-shock amplifies through the
network well beyond the directly-shocked bank. Pairing this with the
stage 3 spillover index lets a researcher distinguish *market-side*
connectedness (asset return co-movement) from *balance-sheet-side*
connectedness (claims-graph contagion).

## API consistency notes

These are flagged for the operator's later review. **No refactor is
performed in this PR.** The four entry points are usable today (modulo
the lanes A/B status above) — these are stylistic inconsistencies that
would matter if the four stages are ever exposed under a single
orchestration surface.

1. **Function vs class entry point.** Stages 1, 2 and 4 are plain
   functions (`score_event`, `replay_macro_events`, `debt_rank`); stage 3
   is a dataclass-style indicator (`DYSpilloverIndicator`, instantiated
   then `.compute(data)`).
   - `cbsrm/macro/macro_events.py:237` — `def score_event(...)`
   - `cbsrm/indicators/dy_spillover.py:168-169` — `@dataclass` /
     `class DYSpilloverIndicator`

2. **Return shape: dict vs `IndicatorResult` dataclass.** `score_event`
   returns a plain `dict[str, Any]` with ad-hoc keys; `DYSpilloverIndicator.compute`
   returns an `IndicatorResult` (`values: pd.Series`, `metadata: dict`,
   `indicator_id`, `version`). A composer that wants to ingest both
   stages today has to special-case the return type.
   - `cbsrm/macro/macro_events.py:237` (`-> dict[str, Any]`)
   - `cbsrm/indicators/dy_spillover.py:187` (`-> IndicatorResult`)

3. **Date typing not enforced.** `score_event` does not carry a release
   date at all (caller's responsibility); the planned `replay_macro_events`
   expects an `events_df` with a `date` column whose type is operator-
   chosen (string vs `pd.Timestamp`); `DYSpilloverIndicator.compute`
   uses `cleaned.index[-1]` and accepts whatever the input index type is.
   No consistent date-handling convention across stages.
   - `cbsrm/macro/macro_events.py:237-300` — no date parameter
   - `cbsrm/indicators/dy_spillover.py:211` — `index=[cleaned.index[-1]]`

4. **Scale / threshold parameters are positional defaults, not a config
   object.** `score_event` accepts `in_line_band_z` as a float kwarg with
   default 0.25; `DYSpilloverIndicator` carries `lag_order=4` and
   `forecast_horizon=10` as dataclass fields. Both are valid; they are
   inconsistent.
   - `cbsrm/macro/macro_events.py:244` (`in_line_band_z: float = 0.25`)
   - `cbsrm/indicators/dy_spillover.py:180-181` (`lag_order: int = DEFAULT_LAG_ORDER`,
     `forecast_horizon: int = DEFAULT_FORECAST_HORIZON`)

5. **Identifier / version metadata is not present on the stage 1 output.**
   `IndicatorResult` carries `indicator_id` and `version`; `score_event`'s
   returned dict carries only `event` and the embedded registry `spec`.
   If the four stages are ever audited through the same chain, stage 1
   would need a wrapper to emit `indicator_id="MACRO-EVENT-SCORER"` +
   version.
   - `cbsrm/indicators/dy_spillover.py:172-173` (`id: str = "DY-SPILLOVER"`,
     `version: str = "1.0.0"`)
   - `cbsrm/macro/macro_events.py:347-363` — no `indicator_id` / `version`

6. **Naming style: snake_case throughout for keys, but `id` (stage 3) vs
   `indicator_id` (in `IndicatorResult`).** Minor; just flagging.
   - `cbsrm/indicators/dy_spillover.py:172` (`id: str = ...`) vs
     `cbsrm/indicators/dy_spillover.py:190` (`indicator_id=self.id`).

7. **Severity / classification vocabularies differ.** `score_event`
   emits a 5-level severity ladder
   (`trivial|mild|moderate|large|extreme`); `DYSpilloverIndicator` emits
   a 4-level classification
   (`LOW_COUPLING|MODERATE_COUPLING|ELEVATED_COUPLING|HIGH_CONTAGION`).
   Casing differs too (lowercase vs SCREAMING_SNAKE_CASE).
   - `cbsrm/macro/macro_events.py:189-195` (severity ladder)
   - `cbsrm/indicators/dy_spillover.py:213-220` (classification bands)

None of these block the v0.8 flow. They are pre-flagged for an eventual
unification pass — likely a `cbsrm.composer` layer that adapts each
stage's native return into a single `PipelineRecord` shape.

## What's NOT in this version

The v0.8 flow is intentionally a research narrative, not a production
pipeline. The following are out of scope for v0.8:

- **Live consensus feed.** Stage 1 expects the caller to supply `actual`
  and `consensus` from a release calendar. CBSRM ships no Trading-Economics
  / Refinitiv / Bloomberg consensus adapter.
- **Multi-jurisdiction integrator.** The flow as documented is single-
  jurisdiction (US prints, US-asset price panels, US-bank network). A
  cross-jurisdiction integrator (EUR / GBP / JPY events propagating into
  USD-asset connectedness) is roadmapped for a later release.
- **Streaming / real-time orchestration.** Each stage is invoked
  imperatively in research notebooks. There is no scheduled job, no
  WebSocket source, no FastAPI endpoint that strings the four stages
  together.
- **Auto-calibrated bank-claims matrix.** Stage 4 expects the caller to
  supply `L`, `E`, `h0`. CBSRM does not yet ship a builder that
  constructs `L` from BIS consolidated banking statistics — that is on
  the v0.9+ roadmap.
- **Backtesting harness.** Replaying stage 1-4 across the canonical
  CRISIS_WINDOWS (2008Q4, 2020Q1, 2023Q1) is planned as a separate
  notebook deliverable, not part of v0.8 core.
- **GUI / dashboard wiring.** The Streamlit demo (`dashboard/streamlit_app.py`)
  surfaces v0.5 risk-pricing readings; it does not yet visualise the
  stage 1-4 flow.

The boundary is deliberate: v0.8 establishes the *vocabulary* (macro
event → replay → spillover → DebtRank) and makes each stage independently
runnable. Composition across stages is left as a user-driven research
loop until the v0.9 composer layer ships.
